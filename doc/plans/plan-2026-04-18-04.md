# Plan 06 — Playback FSM (Transport + Intents)

## Status (2026-04-19)

Tasks 1, 2, 4, 5, 6, 7 shipped on `sprint/playback-fsm`. **Task 3
(`PlaybackEngine` wiring) is deferred to Plan 07** (reverse-path
unification). Rationale: the FSM module and its property suite are
independently valuable and ship cleanly on their own; wiring through
`PlaybackEngine` requires coordinated changes to the same call-sites
that Plan 07 rewrites for reverse-path unification, so bundling them
avoids a double-touch. The FSM carries narrow
`#[allow(dead_code)]` attributes on `Input`/`MixerCommand`/`PlaybackFsm`
until Plan 07 consumes them.

Q7 was revised during implementation — the original "Restart forces
Playing" formulation produced `(Restart, Pause)` counterexamples in
the property suite. The shipped Q7 is "Restart from Stopped is a
no-op", matching `PlaybackEngine::restart_program`'s early-return
semantics. See the Q7 section below.

## Goal

Formalize the UI-observable playback state (`{Stopped, Playing, Paused}` plus
loop/reverse flags and pending `seek`/`restart` intents) as an explicit
`rust-fsm` state machine, validated with a `proptest-state-machine` suite.
The mixer-thread real-time logic (frame iteration, crossfade ramps,
scratch-buffer scheduling) stays as-is; the FSM is the coordinator that
the `PlaybackEngine::set_*` methods and the TUI's playback actions all
dispatch through.

## Context

Playback is the second of three regions being ported to `rust-fsm`
(markers → playback → TUI). Motivation captured in
`doc/designs/debt-playback.md`:

- Tri-state `PlaybackState` is implicit — set via `AtomicU8` writes from
  multiple call-sites with no single transition table.
- Reverse playback has two independent code paths (pre-reverse buffer
  copy + runtime atomic flag) that can compose incorrectly. Unification
  is flagged as a prerequisite for clean FSM modelling but is **out of
  scope for this sprint** (Plan 07 picks it up after the FSM is
  established).
- `pending_seek` / `pending_restart` / `loop_enabled` are ad-hoc atomics
  toggled from the UI; their interaction with playback state is not
  covered by any invariant.

Current call-site surface (`src/engine/playback.rs`, 1703 LOC):
- `PlaybackState` enum — `Stopped | Playing | Paused`
- `SourceControl { frame, pending_seek, pending_restart, loop_enabled,
  reversed }` — read by the mixer thread
- `PlaybackEngine::{play_with_segments, pause, resume, stop, seek_to,
  restart, set_reversed, set_loop_enabled}` — UI-facing mutators

Coverage today: ~30 unit tests around `SegmentSource`, zero proptests.

## Tasks

### 1. FSM module `src/engine/playback_fsm.rs`

**Problem:** Playback state is ad-hoc; transitions are scattered across
UI action handlers and mixer-thread observers.

**Solution:** A `StateMachineImpl` with:

- **State struct:**
  ```rust
  pub struct PlaybackFsmState {
      pub transport: Transport,    // Stopped | Playing | Paused
      pub reversed: bool,          // runtime reverse toggle
      pub loop_enabled: bool,      // auto-restart program on end
      pub pending_seek: Option<u32>,
      pub pending_restart: bool,
  }
  pub enum Transport { Stopped, Playing, Paused }
  ```
- **Input enum:** `Play`, `Pause`, `Resume`, `Stop`, `Seek(u32)`,
  `Restart`, `ToggleReverse`, `SetReverse(bool)`, `ToggleLoop`,
  `SetLoop(bool)`, `SegmentEnded`, `ProgramEnded`, `ConsumeSeek`,
  `ConsumeRestart`.
- **Output enum:** side-effect descriptors (`StartMixer`, `StopMixer`,
  `PauseMixer`, `ResumeMixer`) — lets tests assert on the rodio
  interactions without running real audio.
- **Transition rules:**
  - `Play` from `Stopped` → `Playing` + `Output::StartMixer`.
  - `Pause` from `Playing` → `Paused` + `Output::PauseMixer`.
  - `Resume` from `Paused` → `Playing` + `Output::ResumeMixer`.
  - `Stop` from any transport → `Stopped` + `Output::StopMixer` +
    clear `pending_*`.
  - `Seek(p)` in any transport → `pending_seek = Some(p)`.
    `ConsumeSeek` (dispatched by mixer) clears it.
  - `Restart` → `pending_restart = true`. Valid in any transport; mixer
    consumes via `ConsumeRestart` which clears the flag and snaps
    transport to `Playing` (matches today's `restart()` behavior).
  - `ProgramEnded`: if `loop_enabled`, sets `pending_restart = true`
    and stays `Playing`; else transitions to `Stopped`.
  - `ToggleReverse` / `SetReverse` — flips `reversed` (or assigns).
    Allowed in any transport.

**API surface:**
- `pub struct PlaybackFsm { machine: StateMachine<...> }`
- `pub fn consume(&mut self, input: Input) -> Option<Output>`
- `pub fn state(&self) -> &PlaybackFsmState`
- Legacy accessors: `transport()`, `reversed()`, `loop_enabled()`.

### 3. Wire through `PlaybackEngine`

**Problem:** UI calls `PlaybackEngine::set_reversed(bool)`,
`set_loop_enabled(bool)`, `seek_to(u32)`, `restart()`, `pause()`,
`resume()`, `stop()`. Each mutates atomics directly.

**Solution:** Add `PlaybackEngine::fsm: Mutex<PlaybackFsm>` (or
equivalent — the TUI-facing API is single-threaded). Each `set_*`
method becomes:

```rust
pub fn set_reversed(&mut self, v: bool) {
    let out = self.fsm.lock().consume(Input::SetReverse(v));
    // apply atomics from new fsm state
    self.control.reversed.store(
        self.fsm.lock().state().reversed,
        Ordering::Relaxed,
    );
}
```

The mixer thread keeps reading the atomics as today. The FSM is the
single source of truth for UI-observable transitions; atomics are the
wire.

**API surface:** No external change — `PlaybackEngine`'s public methods
are unchanged. Internal routing flows through the FSM.

### 4. Property-test layout `tests/engine/playback_fsm/`

**Problem:** Zero proptests today for the transport/intent state
machine.

**Solution:** Four-file layout matching the marker_fsm pattern:

- `generators.rs` — `any_input(state)` + restricted streams
  (e.g. `no_stop_toggle` for fixed-point tests).
- `prop.rs` — `ReferenceStateMachine for PlaybackFsmModel` + each
  property test. Properties below.
- `unit.rs` — migrated unit tests for `SegmentSource::pending_seek`,
  `pending_restart`, etc. that currently live in playback.rs's inline
  `#[cfg(test)] mod`.
- `test.rs` — entrypoint + `TestConfig` (cases, verbose, env-var
  override) and a `pub const MAX_STEPS: usize = 32` referenced by
  both the `prop_state_machine!` sequential ranges and the doc.
  Default cases=64.

### 5. Properties (must pass)

| #   | Property                                                                                               |
|-----|--------------------------------------------------------------------------------------------------------|
| Q1  | `Play ∘ Stop ∘ Play ≡ Play` (state reaches Playing regardless of intermediate)                          |
| Q2  | `Pause ∘ Resume ≡ id` when initial transport is Playing                                                 |
| Q3  | `Stop` clears `pending_seek` and `pending_restart` always                                               |
| Q4  | `ToggleReverse ∘ ToggleReverse ≡ id` (boolean idempotence)                                              |
| Q5  | `Seek(p)` followed by `ConsumeSeek` leaves `pending_seek == None` regardless of intermediate inputs     |
| Q6  | `ProgramEnded` with `loop_enabled=true` keeps transport Playing and sets `pending_restart`             |
| Q7  | `Restart` from `Stopped` is a no-op (no `pending_restart` queued against an idle sink)                  |
| Q8  | No-op insertion: `[Pause, Resume]` in a stream preserves end state (when start transport is Playing)    |

Q7 captures today's `PlaybackEngine::restart_program` early-return:
dispatching `Restart` while transport is `Stopped` must not queue a
pending restart intent (otherwise the next `Play` would silently
restart a fresh program mid-startup). During implementation an earlier
"Restart forces Playing" formulation was discarded when the property
suite found `(Restart, Pause)` counterexamples — Restart just queues
the intent, Pause is still valid, and `ConsumeRestart` (fired by the
mixer when it actually restarts) is what flips transport to Playing.

### 6. `TestConfig` struct

Close to the `marker_fsm` pattern, but with the step bound hoisted
to a `const` because `prop_state_machine!`'s sequential range must
be a literal/const expression — it can't read a runtime struct field.
So: `struct TestConfig { cases: u32, verbose: bool }` with
`pub const MAX_STEPS: usize = 32` in
`tests/engine/playback_fsm/test.rs`, used by every
`sequential 1..MAX_STEPS` invocation so bumping the step bound flows
to all harnesses in one edit. Default `cases=64`. Read env
`RIFFGREP_PROPTEST_CASES` if set.
