# Plan 05 — Markers FSM, Data-Mutation Dispatch (Task 5c)

## Goal

Close out the markers FSM carve-out by removing
`PreviewData.markers: Option<MarkerConfig>` and routing every
data-mutation path in `src/ui/mod.rs` through `MarkerFsm`
dispatches instead of in-place mutation + the
`sync_fsm_from_preview` bridge. The FSM becomes the single source
of truth for marker data; `preview.markers` goes away entirely.

## Context

Plan 03 (`plan-2026-04-18-02.md`, PR #17) shipped the FSM module
and property suite. Plan 04 (`plan-2026-04-18-03.md`, PR #18)
carved the four bare scalar fields (`selected_marker`,
`active_bank`, `bank_sync`, `markers_visible`) off `App` and added
`Input::LoadConfig` + the `sync_fsm_from_preview` helper as a bridge
while the marker *data* stayed on `PreviewData.markers`.

Plan 04's Status block explicitly deferred this sprint: "Task 5c
(deferred) requires rewriting `set_marker`, `clear_nearest_marker`,
`clear_bank_markers`, `nudge_marker`, `snap_zero_crossing`,
`marker_reset`, `adjust_rep`, `toggle_infinite_loop`, and CSV
export/import to dispatch through FSM inputs rather than mutating
`preview.markers` in place. ~40 bodies plus ~23 read-site updates.
Sized at a dedicated follow-up sprint (Plan 05) so the incremental
value of 4b ships without that risk."

After this sprint most of the narrow `#[allow(dead_code)]` on
`marker_fsm::Input` data-mutating variants comes off. Three
variants — `NudgeForward`, `NudgeBackward`, and `MarkerReset` —
stay marked as unused because the UI intentionally dispatches via
`SetSelectedMarker` (ZC-resolved) and `LoadConfig` (App-built
reset) instead. Those three variants remain in the FSM API as
surface for future callers and are covered by the proptest suite.
See the Deferred section for rationale.

## Tasks

### 1. Rewrite each mutation method to dispatch through FSM

Targets in `src/ui/mod.rs`:
- `set_marker(index)` → `Input::SetMarker1/2/3(snapped)`
- `clear_nearest_marker()` → `Input::ClearNearestMarker(cursor)`
- `clear_bank_markers()` → `Input::ClearBankMarkers`
- `nudge_marker(forward, n)` → zero-crossing-aware: App resolves
  the ZC target then dispatches `Input::SetSelectedMarker(zc)`.
  (The FSM still carries `Input::NudgeForward`/`NudgeBackward` as
  API surface for a future delta-nudge caller.)
- `snap_zero_crossing(forward)` → `Input::SetSelectedMarker(zc)`
- `marker_reset()` → `Input::LoadConfig(config)` with a freshly
  constructed SOF+EOF `MarkerConfig`. (`Input::MarkerReset { sof,
  eof }` remains in the FSM API for a future caller that wants
  the FSM itself to build the config.)
- `adjust_rep(delta)` → `Input::IncrementRep` / `DecrementRep`
- `toggle_infinite_loop()` → dedicated `Input::ToggleInfiniteLoop`
- CSV export/import → consume `Output::WriteCsv(PathBuf)` /
  `Output::ReadCsv(PathBuf)` from `Input::ExportMarkersCsv` /
  `ImportMarkersCsv`

Each method becomes a thin wrapper: compute the payload (e.g.
snap-to-zero-crossing, cursor position), dispatch the Input, set
the status message. No more in-place `preview.markers` mutation.

### 2. Remove `PreviewData.markers`

Drop the field. Replace read-sites (`preview.markers.as_ref()`,
`bank_a`/`bank_b` accesses) with `self.marker_fsm.config()` calls.
Roughly 23 call-sites per Plan 04's count.

### 3. Remove the `sync_fsm_from_preview` bridge

With `PreviewData.markers` gone there's nothing to sync from.
Delete the helper and every call-site.

### 4. Remove `#[allow(dead_code)]` from `marker_fsm.rs`

Most data-mutating Input variants now have callers. Three
variants remain unwired but are kept as FSM API surface (covered
by the proptest suite): `NudgeForward`, `NudgeBackward`, and
`MarkerReset`. See "Deferred" for rationale. Their narrow
`#[allow(dead_code)]` attrs stay; remove the rest.

### 5. Migrate unit tests

App-level tests that poke `preview.markers` directly need to
install state via `load_similarity_snapshot`-equivalent or via
FSM inputs. Handful of tests; mostly mechanical.

## Verification

App-level behavior (no more dual-source-of-truth between FSM and

## Deferred

- **Delta-nudge FSM dispatch.** `nudge_marker` resolves the
  zero-crossing target App-side and dispatches
  `Input::SetSelectedMarker(zc)`, so `Input::NudgeForward` /
  `NudgeBackward` have no live caller. They stay in the FSM API
  (covered by the proptest suite) behind `#[allow(dead_code)]`
  for a future caller that wants delta-based nudging without ZC
  resolution.
- **FSM-built reset config.** `marker_reset` builds the
  SOF+EOF `MarkerConfig` App-side and dispatches
  `Input::LoadConfig(config)`, so `Input::MarkerReset { sof, eof
  }` has no live caller. Kept as FSM API for a future caller
  that wants the FSM itself to construct the reset config.
