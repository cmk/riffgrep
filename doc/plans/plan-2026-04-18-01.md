## Part A — Validation (user-run, one-time)

The agent can script around these steps, but execution needs the user's
machine: CLAP checkpoint (~600MB), real audio library, GPU or patience.

1. **Install Python deps.** The heavy ML stack lives in opt-in extras —
   a plain `uv sync` only pulls `numpy`. To run the test suite add
   `--extra dev` (pytest + hypothesis). For real inference you want
   the `ml` umbrella plus dev:

   ```
   uv sync --extra ml --extra dev
   ```

   Pulls `laion-clap`, `faiss-cpu`, `librosa`, `soundfile`, `tqdm`, plus
   transitively `torch` / `numba` / `llvmlite`. The `override-dependencies`
   pins in `pyproject.toml` (numba/llvmlite caps + the torch/
   torchvision/transformers range) are tuned to keep this install
   wheel-only on macOS x86_64 + Python 3.12 — no source builds
   expected. If you're on a different platform where those pins still
   miss a wheel and llvmlite falls back to a source build, you'll
   need `cmake` + `llvm@20` via brew; but under the documented target
   this shouldn't happen.

2. **Download the CLAP checkpoint** to
   `~/Library/Application Support/riffgrep/models/music_audioset_epoch_15_esc_90.14.pt`
   (or anywhere, set `LAION_CLAP_CHECKPOINT` to point at it).

3. **Confirm index is populated.** If not, run
   `rfg --index <root>` on a representative subset first (a few
   thousand files is enough for the smoke test).

4. **Small-batch encode.** Exercise the pipeline end-to-end on 500
   files before committing to a full pass:

   ```
   python scripts/embed_encode.py --limit 500
   ```

5. **Small-batch train.** Build the first real codebook:

   ```
   python scripts/embed_train.py --n-train 500
   ```

   Note: FAISS PQ wants ≥256 training samples per subquantizer for
   full-quality clustering; 500 is below that for some sub-spaces but
   adequate for a smoke test. Scale up before production use.

6. **Ranking smoke.** Pick a known kick sample, query, verify the
   top-10 is dominated by kicks:

   ```
   rfg --similar /path/to/known_kick.wav --limit 10
   ```

   Repeat for a snare, a pad, a bass one-shot. Qualitative check — if
   results look reasonable, proceed to full pass.

7. **Full pass.** Drop `--limit` on both scripts and re-run.

If the ranking in step 6 is garbage, stop and debug before the full
pass. The most likely failure modes are: checkpoint mismatch (wrong
LAION-CLAP variant), dtype drift in the preprocess pipeline, or
codebook training starvation. Each has a follow-up test already in
place (P1.1, P1.4, P1.3 respectively).

## Part B — Follow-ups from `doc/reviews/review-0010.md`

### T1. Memory-bounded training-vector fetch (review follow-up #4)

**Problem.** `scripts/embed_train.py::_fetch_training_vectors` does
`SELECT id, embedding FROM samples WHERE ...` with no `LIMIT`, then
samples `n_train` in Python. At 1.2M rows × 2048 bytes = ~2.5 GB held
in Python memory before any training starts. The same shape is in
`embed_encode.py::encode_rows` (`rows = list(rows)`), which also
materializes all candidate IDs; smaller per-row but still 1.2M tuples.

**Solution.** Stream IDs first (a few hundred MB at most), sample the
IDs with `numpy.random.default_rng(seed).choice`, then fetch only the
sampled BLOBs in batches. Preserves reproducibility because the seed
feeds numpy, not SQLite.

**API.** `_fetch_training_vectors` signature unchanged; internals:

```python
ids = [r[0] for r in conn.execute("SELECT id FROM samples WHERE ...")]
picks = rng.choice(len(ids), size=min(n_train, len(ids)), replace=False)
# batch-fetch embedding for sampled ids, yield rows in one (N, 512) array
```

For `encode_rows` the situation is lighter: its input is a `(int,
str)` tuple stream that's a few hundred MB at 1.2M scale, not
multiple GB. We keep `_select_rows` returning a list and drop the
redundant extra `list(rows)` conversion — true streaming would also
break the tqdm total count, which is useful UX. Memory characteristic
after this sprint is `O(candidates × row_tuple)` in encode and
`O(n_train × EMBED_BYTES)` in train — the latter is the one that
matters at 1.2M scale.

### T2. FAISS centroid layout spot-check (review follow-up #5)

**Problem.** `test_faiss_codebook_matches_layout` only asserts that
decoded centroids are finite. A transposition bug (M ↔ K swap) would
pass the size + finiteness checks but break Rust's decoder.

**Solution.** After `train()` returns a codebook blob, use FAISS's own
`ProductQuantizer.decode` to decode an all-zeros code (centroid 0 of
every subquantizer), and assert the Python-side layout produces a
matching vector when indexed at `[m, 0, :]` for all m.

**File.** `scripts/tests/test_codebook_rust_compat.py` — extend
`test_faiss_codebook_matches_layout`.

### T3. Skip logging in `encode_rows` (review follow-up #6)

**Problem.** When `preprocess()` returns `None`, `encode_rows` silently
drops the row and moves on. For a 1.2M-file run, accumulated silent
skips are invisible — no way to tell "5 failures" from "500k failures"
from the stdout summary.

**Solution.** Count `None` returns per batch and print a one-line
total to stderr at the end of `encode_rows` when `skipped > 0`. The
tqdm bar is left alone — a tail summary is enough operational signal
and avoids the UX jank of a `postfix` that updates mid-run.

**API.** Keep the return shape as `int` (committed count). Print to
stderr a one-liner before returning when `skipped > 0`.

### T4. Hypothesis property tests for `embed_preprocess` (review follow-up #8)

**Problem.** CLAUDE.md mandates property-based testing for modules that
parse, encode, or transform data. `embed_preprocess.py` is a pure
data-transform module. All existing tests are example-based.

**Solution.** New `scripts/tests/test_preprocess_properties.py` using
`hypothesis`. Three properties:

- **Output length invariant.** For any synthetic float32 array whose
  trimmed length is ≥1 sample, `preprocess(...).shape[0] ==
  target_sr * window_seconds`.
- **Peak normalization invariant.** After preprocess,
  `np.max(np.abs(out))` is within ±1 dB of `_db_to_amp(PEAK_DB)`
  (unless the input was all-silent, in which case the return is
  `None`).
- **All-silence returns None.** Inputs where every sample is below
  `_db_to_amp(SILENCE_DB)` return `None`.

**Dependencies.** Add `hypothesis>=6.0` to the `dev` extra.
