## Goal

Formalize the search-transport + similarity-mode state machine that
today lives as ad-hoc flags on `App` (`search_in_progress`,
`search_pending`, `query_changed`, `in_similarity_mode`,
`similarity_results`), and introduce the runner / effects-as-data
pattern that stdio will embed. First sub-region of the TUI refactor
(markers → playback → TUI; TUI fragments into N sub-FSMs per
`stdio/doc/designs/tui-fsm.md`).

## Context

Per `stdio/doc/designs/tui-fsm.md`, the long-term goal is to embed
riffgrep's TUI as a "form" inside stdio ("Claude Code for studios").
Six design considerations drive this plan:

1. **Serializable state** — `SearchFsmState` derives
   `Serialize + Deserialize`.
2. **Typed events** — `Input` enum is the external API boundary.
3. **Effects-as-data** — `Output` is a pure effect enum
   (`SpawnSearch`, `CancelSearch`, `FilterSimilarity`,
   `FireSelection`); a separate runner consumes outputs and performs
   I/O.
4. **Keybindings above** — `crossterm::KeyEvent → Action → Input` is a
   two-step pipeline. The FSM sees only typed `Input`.
5. **Typed actions at the output edge** — `FireSelection` is a
   placeholder; `SearchRunner::dispatch` reads wrapper state and
   synthesizes `TypedAction::LoadSample(PathBuf)`.
6. **Explicit cancel/cleanup** — the FSM/runner boundary is
   effects-as-data, so `SearchRunner` does **not** own tokio task
   handles. `Output::CancelSearch` is emitted by transitions like
   `EnterSimilarityMode`; the App-side integration honors it by
   aborting/draining its own `JoinHandle`. The runner signals back
   via `Input::SearchCancelled` / `SearchFailed` / `SearchSettled`.

### Current state surface

From `src/ui/mod.rs`:

- `search_in_progress: bool` — a remote search is active.
- `search_pending: bool` — a new search has been dispatched but the
  first batch hasn't arrived (keeps old results visible, prevents
  flicker).
- `query_changed: bool` — debounce flag: set on every query keystroke,
  consumed by the debounce tick.
- `in_similarity_mode: bool` — alternate dispatch: typing filters a
  snapshot locally instead of re-querying the DB.
- `similarity_results: Option<Vec<TableRow>>` — immutable snapshot for
  local filter restoration when the query clears.

The dispatch fork at `src/ui/mod.rs:3011–3024` is the key integration
point: on each debounce tick, branch on `in_similarity_mode` to either
call `filter_similarity_results()` (local) or cancel+respawn the DB
search (remote).

Similarity is therefore **not a parallel state space** — it's a mode
variant of the search transport. One unified FSM with a mode dimension
fits the model; three sub-FSMs would share 80% of their state and split
the debounce-fork awkwardly.

Coverage today: similarity mode was the bug cluster in PR #14
(SPRINT `3e35cd0` + round fixes); zero proptests.

## Tasks

### 1. FSM module `src/engine/search_fsm.rs`

**Problem:** Search state is ad-hoc flags with no transition table.

**Solution:** `StateMachineImpl` with:

- **State struct:**
  ```rust
  #[derive(Serialize, Deserialize, ...)]
  pub struct SearchFsmState {
      pub transport: Transport,       // Idle | Pending | Running | Settled
      pub mode: Mode,                 // Remote | Similarity
      pub query: String,
      pub debounce_dirty: bool,       // matches today's query_changed
  }
  pub enum Transport { Idle, Pending, Running, Settled }
  pub enum Mode { Remote, Similarity }
  ```
  Note: `results`, `selected`, `scroll_offset`, `similarity_results`
  snapshot, and `columns`/`sort` live on the **runner**, not the FSM.
  Per the rust-fsm gotcha in `tui-fsm.md` §rust-fsm: "the FSM governs
  modes, not mutations" — appending results is a wrapper method, not
  an FSM input.

- **Input enum (external events):**
  - User: `QueryChanged(String)`, `QueryCleared`, `SubmitQuery`,
    `FireSelection`. (Selection navigation is a wrapper-method data
    mutation — `SearchRunner::move_selection` — not an FSM input, per
    the "modes not mutations" rule.)
  - Timing: `DebounceTick`.
  - External (from runner/mixer equivalent):
    `SearchStarted`, `SearchSettled { total: usize }`,
    `SearchCancelled`, `SearchFailed`.
  - Mode control: `EnterSimilarityMode`, `ExitSimilarityMode`.

- **Output enum (effect descriptors):**
  - `SpawnSearch { query: String }`
  - `CancelSearch`
  - `FilterSimilarity { query: String }`
  - `FireSelection` — placeholder; `SearchRunner::dispatch` reads
    wrapper state and translates it into
    `TypedAction::LoadSample(PathBuf)`.

- **Transition rules** (core invariants; see §Properties for the full
  list):
  - `QueryChanged(q)` while `mode=Remote`: update `query`, set
    `debounce_dirty=true`, no transport change yet.
  - `QueryChanged(q)` while `mode=Similarity`: update `query`, set
    `debounce_dirty=true`, no transport change (filtering is debounced
    on the same path as remote — the subsequent `DebounceTick` emits
    `Output::FilterSimilarity`). Matches today's TUI debounce
    behavior; avoids re-filtering on every keystroke.
  - `DebounceTick` while `debounce_dirty && mode=Remote`:
    transport → Pending, clear `debounce_dirty`, emit
    `Output::SpawnSearch`. (rust-fsm's `output()` returns a single
    `Option<Output>`; the "cancel + spawn" pair is encoded in
    `Output::SpawnSearch`'s runner contract — the runner aborts any
    in-flight handle before starting the new one. No separate
    `CancelSearch` fires on tick.)
  - `DebounceTick` while `debounce_dirty && mode=Similarity`:
    emit `Output::FilterSimilarity`, clear `debounce_dirty`.
  - `SearchStarted`: Pending → Running.
  - `SearchSettled`: Pending | Running → Settled. (Pending can
    short-circuit when a cancel arrives before the first batch.)
  - `SearchCancelled`: any → Settled (the caller then decides whether
    to Idle or Pending).
  - `EnterSimilarityMode`: mode → Similarity, transport → Settled,
    emit `Output::CancelSearch` (any in-flight remote search is moot).
  - `ExitSimilarityMode` while `mode=Similarity`: mode → Remote,
    transport → Idle, `debounce_dirty=true` (forces a fresh remote
    search). **Guarded no-op when already `mode=Remote`** — without
    this guard a defensive call would clobber a live transport. See
    the `exit_similarity_from_remote_is_noop` inline test.

**API surface:**
- `pub struct SearchFsm { machine: StateMachine<SearchMachine> }`
- `pub fn consume(&mut self, input: Input) -> Option<Output>`
- `pub fn state(&self) -> &SearchFsmState`
- Legacy accessors matching today's App reads (`search_in_progress()`,
  `in_similarity_mode()`, `query()`).

### 2. Runner module `src/engine/search_runner.rs`

**Problem:** `App` today owns tokio `JoinHandle`s, the similarity
snapshot, and the results list, and performs I/O directly from key
handlers. stdio's embedding contract needs a substitution seam.

**Solution:** `SearchRunner` type owning:
- `fsm: SearchFsm` (pure state).
- `similarity_snapshot: Option<Vec<TableRow>>`.
- `results: Vec<TableRow>`, `selected: usize`, `scroll_offset: usize`,
  `total_matches: usize`.
- `columns`, `sort_column`, `sort_ascending` (remain under runner for
  now; may move to their own sub-FSM later).

**No `JoinHandle`, no tokio spawn.** Per `stdio/doc/designs/tui-fsm.md`
§3 ("side effects returned, not performed"), the runner's job is to
delegate to the FSM, mutate its own data fields, and return the
emitted `Output` + a synthesized `TypedAction` (when applicable).
The App-side integration in Task 4 owns the tokio tasks and honors
the returned `Output`.

```rust
impl SearchRunner {
    pub fn dispatch(&mut self, input: Input) -> DispatchResult {
        let output = self.fsm.consume(input);
        let action = match &output {
            Some(Output::FireSelection) => {
                self.selected_path().map(TypedAction::LoadSample)
            }
            _ => None,
        };
        DispatchResult { output, action }
    }
    // Plus setter/accessor wrapper methods — set_results,
    // append_results, set_total_matches,
    // load_similarity_snapshot, apply_similarity_filter, etc.
}
```

`TypedAction` is a new typed enum replacing today's stringified
action dispatches at the output edge.

### 3. Keybinding extraction

**Problem:** `src/ui/actions.rs` conflates key → action mapping with
action → App-method dispatch.

**Solution:** Split into two pure functions:
- `crossterm::KeyEvent → Option<Action>` (config-driven, already
  partially in `actions::Keymap`).
- `Action → Option<search_fsm::Input>` (new adapter layer).

The second function is the FSM's external API; the first is the
keymap, which stdio can swap out for form-protocol events.

### 4. App integration

**Problem:** `App` today owns all state and does all dispatch.

**Solution:** Minimal first-pass: `App.search_runner: SearchRunner`,
with the 5 bare search-state fields (`search_in_progress`,
`search_pending`, `query_changed`, `in_similarity_mode`,
`similarity_results`) carved onto the runner and read through
accessors. Behavior-preserving; follows the markers carve-out
pattern from Plan 04.
