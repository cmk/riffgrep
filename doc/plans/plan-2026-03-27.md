# Embedding Phase 1: Rust Scaffolding

## Goal

Wire up the full similarity search UX (DB schema, TUI column, sort action,
CLI flag) using synthetic embeddings. No CLAP model, no Python. Everything
testable with random or hand-crafted f32 vectors injected into the DB.

---

## Testing Strategy

### Synthetic embeddings for testing

Inject known vectors into the DB in unit tests. This lets us verify
search correctness without any model dependency:

```rust
// Three files: A=[1,0,0,...], B=[0.9,0.1,0,...], C=[0,0,1,...]
// Query=A ‚Üí ranking should be A(1.0), B(~0.99), C(~0.0)
```

### Test categories

1. **Schema tests** ‚Äî migration adds column, NULL default, BLOB round-trip
2. **Distance tests** ‚Äî L2 distance on known vectors, sim score mapping
3. **Sort tests** ‚Äî subject pinned to top, ascending distance, NULL‚Üíbottom
4. **TUI tests** ‚Äî `sim` column renders correctly, blank when inactive
5. **CLI tests** ‚Äî `--similar` flag parses, errors on missing embedding
6. **Property tests** ‚Äî sim(x,x)=1.0, sim is symmetric, triangle inequality

### Chunk 2: Similarity search engine

Pure Rust L2 distance computation and result ranking.

**Files:** `src/engine/similarity.rs` (new)

**Changes:**
1. `pub fn l2_distance(a: &[f32], b: &[f32]) -> f32`
2. `pub fn similarity_score(dist: f32, max_dist: f32) -> f32`
   ‚Äî returns `1.0 - (dist / max_dist)`, clamped to [0.0, 1.0]
3. `pub struct SimilarityResult { pub id: i64, pub path: PathBuf, pub dist: f32, pub sim: f32 }`
4. `pub fn search_similar(query: &[f32], candidates: &[(i64, Vec<f32>)], limit: usize) -> Vec<SimilarityResult>`
   ‚Äî brute-force L2, sort ascending, compute sim scores relative to max_dist
5. Subject (query file) is always inserted at position 0 with sim=1.0

**Tests:**
- `test_l2_distance_zero` ‚Äî identical vectors ‚Üí 0.0
- `test_l2_distance_known` ‚Äî [1,0] vs [0,1] ‚Üí sqrt(2)
- `test_l2_distance_symmetric` ‚Äî dist(a,b) == dist(b,a)
- `test_similarity_score_identity` ‚Äî dist=0, max_dist>0 ‚Üí 1.0
- `test_similarity_score_max` ‚Äî dist=max_dist ‚Üí 0.0
- `test_search_similar_ordering` ‚Äî close vectors rank higher
- `test_search_similar_subject_at_top` ‚Äî query file always position 0, sim=1.0
- `test_search_similar_empty_candidates` ‚Äî returns just the subject
- **Proptest:** `proptest_l2_nonnegative` ‚Äî dist >= 0 for all vectors
- **Proptest:** `proptest_l2_symmetric` ‚Äî dist(a,b) == dist(b,a) for random vectors
- **Proptest:** `proptest_l2_triangle_inequality` ‚Äî dist(a,c) <= dist(a,b) + dist(b,c)
- **Proptest:** `proptest_sim_in_range` ‚Äî sim always in [0.0, 1.0]
- **Proptest:** `proptest_sim_monotonic` ‚Äî closer distance ‚Üí higher sim

### Chunk 4: TUI similarity sort

### Chunk 5: Synthetic embedding injector (test utility)

A small helper for tests and prototyping that injects random or
hand-crafted embeddings into the DB.

**Files:** `scripts/inject_synthetic.py` (or Rust test helper)

For unit tests, a Rust helper:
```rust
fn inject_embedding(db: &Database, path: &str, vector: &[f32; 512])
```

For manual TUI testing, a Python one-liner:
```bash
python3 -c "
import sqlite3, struct, numpy as np
db = sqlite3.connect('index.db')
for row in db.execute('SELECT id, path FROM samples'):
    vec = np.random.randn(512).astype(np.float32)
    vec /= np.linalg.norm(vec)  # L2 normalize
    db.execute('UPDATE samples SET embedding=? WHERE id=?',
               (vec.tobytes(), row[0]))
db.commit()
"
```

This lets us test the full TUI similarity sort flow before CLAP exists.

**Tests:**
- `test_inject_and_search` ‚Äî inject 10 synthetic vectors, verify
  search_similar returns correct ordering

## Execution Order

1. **Chunk 2 first** ‚Äî similarity engine is pure math, fully testable in
   isolation, no DB dependency. Get the distance/ranking logic right.
2. **Chunk 1** ‚Äî schema migration. Depends on nothing, unblocks chunks 3-5.
3. **Chunk 5** ‚Äî synthetic injector. Needed to manually test chunks 3 and 4.
4. **Chunk 3** ‚Äî CLI flag. Uses chunks 1+2.
5. **Chunk 4** ‚Äî TUI sort. Uses chunks 1+2, most complex, do last.

---

## Non-goals for Phase 1

- No CLAP model, no Python encoding pipeline (phase 2)
- No PQ acceleration (phase 3, only needed at >100K scale)
- No `--sounds-like` text query (future)
- No BEXT storage (DB-only per design decision)
