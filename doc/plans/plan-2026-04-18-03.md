## Goal

Finish the markers FSM sprint by carving the four bare marker fields
(`selected_marker`, `active_bank`, `bank_sync`, `markers_visible`) and the
nested `PreviewData.markers: Option<MarkerConfig>` off `App` so that every
read and write of marker state flows through
`crate::engine::marker_fsm::MarkerFsm`.

## Context

Plan 03 (`doc/plans/plan-2026-04-18-02.md`) shipped the FSM module, the
property suite, and the `Bank` enum unification under Task 4a. Task 4b was
explicitly deferred because the scope — ~52 field references in
`src/ui/mod.rs`, 23 `preview.markers` accesses, plus plumbing through
`widgets.rs` and `actions.rs` — was too risky to land at the end of an
autonomous session alongside the FSM implementation itself.

With Plan 03 shipped (PR #17), the FSM module exists, is tested in
isolation, and carries a `#![allow(dead_code)]` waiting for App to consume
its API. This plan removes that allowance.

Current call-site counts (`sprint/markers-app-carve`, base
`sprint/markers-fsm`):

| Field / access           | `ui/mod.rs` | `ui/widgets.rs` | `ui/actions.rs` |
|--------------------------|-------------|-----------------|-----------------|
| `selected_marker`        | 14          | 9 (combined)    | 3 (combined)    |
| `active_bank`            | 22          |                 |                 |
| `bank_sync`              | 12          |                 |                 |
| `markers_visible`        | 4           |                 |                 |
| `preview.markers`        | 23          | 0               | 0               |

Total ~86 field accesses + ~23 data-container accesses. 1014 LOC of
`marker_fsm.rs` provides the replacement API.

## Tasks

Ordered so every commit leaves `cargo test` green and the compiler forces
the next migration when a field is removed.

### 1. Add `marker_fsm: MarkerFsm` field to `App`

**Problem:** `MarkerFsm` has no owner on `App` yet.

**Solution:** Add the field; initialize in `App::new` with
`MarkerFsm::new()`. Leave the four bare fields and `PreviewData.markers`
in place. No call-site changes. Serves as a no-op beachhead.

**API surface:** `App` gains one field.

### 2. Migrate `markers_visible`

**Problem:** The visibility flag has 4 direct field accesses and is read
by the visibility gate in `dispatch_action` (guards marker-edit actions).

**Solution:**
- Remove `markers_visible: bool` from `App`.
- Add `pub fn markers_visible(&self) -> bool` reading
  `self.marker_fsm.markers_visible()`.
- Replace the 2 write sites (`toggle_marker_display`) with
  `self.marker_fsm.consume(Input::ToggleMarkerDisplay)`.
- Replace the 2 read sites with the accessor method.
- Preserve the cosmetic behavior from `toggle_marker_display`: when
  flipping off, the FSM already clears `selected_marker` internally.
  Confirm no App-side state mutation is lost.

**API surface:** `App.markers_visible: bool` → `App::markers_visible() -> bool`.

### 3. Migrate `bank_sync`

**Problem:** 12 call-sites read/write `self.bank_sync`.

**Solution:**
- Remove the field; add `App::bank_sync() -> bool` accessor.
- Replace the 1 write site (`toggle_bank_sync`) with
  `self.marker_fsm.consume(Input::ToggleBankSync)`.
- Replace the 11 read sites with the accessor.

**API surface:** `App.bank_sync: bool` → `App::bank_sync() -> bool`.

### 4. Migrate `active_bank`

**Problem:** 22 call-sites touch `self.active_bank`, including the
`active_bank_ref()` / `active_bank_mut()` helpers (lines 1108, 1117) that
dereference into `preview.markers`.

**Solution:**
- Remove the field; add `App::active_bank() -> Bank` accessor.
- Replace the 1 write site (`toggle_bank`) with
  `self.marker_fsm.consume(Input::ToggleBank)`.
- Update `active_bank_ref()` / `active_bank_mut()` to read
  `self.marker_fsm.active_bank()` instead of the field.
- Replace the remaining ~20 `match self.active_bank` or `self.active_bank == Bank::X`
  sites with the accessor.
- Note: `active_bank_ref/mut` still return into `preview.markers` — the
  data container migration happens in Task 5.

**API surface:** `App.active_bank: Bank` → `App::active_bank() -> Bank`.

### 5. Migrate `preview.markers` into FSM `config`

**Problem:** `PreviewData.markers: Option<MarkerConfig>` owns the marker
data; the FSM's `config: MarkerConfig` field is an unused shadow. The two
must converge before `SelectNext/Prev` can dispatch through the FSM —
`cycle_selection` reads `defined_selections(&self.config)`.

**Solution:**
- Add `Input::LoadConfig(MarkerConfig)` to `marker_fsm::Input`.
  Transition rule: unconditionally replaces `state.config`, returns
  `None` output (idempotent load).
- Remove `markers: Option<MarkerConfig>` from `PreviewData`.
- On preview arrival (`set_preview` / equivalent), dispatch
  `Input::LoadConfig` if BEXT yielded a config. Otherwise leave the FSM
  config as-is from the previous file (document this choice; matches
  today's behavior where `preview.markers` carries forward between files
  until overwritten).
- Replace data-mutating App methods (`set_marker`, `nudge_marker`,
  `clear_nearest_marker`, `clear_bank_markers`, `marker_reset`,
  `increment_rep`, `decrement_rep`, CSV export/import) with dispatch to
  the corresponding `Input::...`. Keep the App-layer responsibility for
  computing inputs that require UI context (e.g. zero-crossing snap
  computes the target sample, then dispatches `Input::SetSelectedMarker`).
- Update BEXT write path to read `self.marker_fsm.config()`.
- Update waveform rendering (`render_waveform_panel` reads
  `preview.markers.as_ref()`) to read `self.marker_fsm.config()`.
- Migrate ~40 test fixtures across `ui/mod.rs`.

**API surface:**
- `marker_fsm::Input` gains `LoadConfig(MarkerConfig)`.
- `PreviewData.markers: Option<MarkerConfig>` removed.
- All data-mutating App methods become thin wrappers around
  `self.marker_fsm.consume(...)`.

**Risk:** This is the largest step. Do it after the scalar fields have
been migrated so most call-sites already funnel through accessors, and
the remaining churn is concentrated on data-container access.

### 6. Migrate `selected_marker`

**Problem:** 14 call-sites; the cycle logic (`select_next_marker`,
`select_prev_marker`) reads defined indices from `preview.markers`.

**Solution (enabled by Task 5):**
- Remove the field; add `App::selected_marker() -> Option<usize>`
  accessor (returning the FSM selection mapped through `as_index()`).
- Replace `select_next_marker` / `select_prev_marker` bodies with
  `self.marker_fsm.consume(Input::SelectNextMarker)` /
  `SelectPrevMarker`, then call `self.seek_to_selected_marker()` as
  today. The FSM's `cycle_selection` now reads the synced config from
  Task 5, so behavior matches the App-version logic.
- Update the 12 read sites.
- Update `ensure_markers()` — the current implementation resets
  `selected_marker` when markers become visible; the FSM does the
  inverse (clears on invisible). Confirm the interaction is preserved.

**API surface:** `App.selected_marker: Option<usize>` →
`App::selected_marker() -> Option<usize>`.

### 7. Cleanup

- Remove `#![allow(dead_code)]` from `src/engine/marker_fsm.rs`.
- Remove unreachable branches surfaced by the full-integration
  (specifically the `seg > 3` guard at line 413 called out in review
  0017).
- Add `sof.min(MAX_MARKER_POS)` clamp in `reset_to_quartiles` per
  review 0017 follow-up.
- Remove any `active_bank_mut()` / `active_bank_ref()` helpers that
  become dead after Task 5.
