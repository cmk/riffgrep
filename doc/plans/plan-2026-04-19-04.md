# Plan 09 — Search FSM, Keybinding Split + App Integration

## Goal

Complete Plan 08's deferred Tasks 3 and 4: split
`src/ui/actions.rs` into a two-stage pipeline
(`crossterm::KeyEvent → Action → search_fsm::Input`) and carve the
five bare search-state fields (`search_in_progress`,
`search_pending`, `query_changed`, `in_similarity_mode`,
`similarity_results`) off `App` onto `SearchRunner`.

## Context

Plan 08 (`plan-2026-04-19-01.md`, PR #21) shipped the pure FSM,
the effects-as-data runner, and the R1–R7 property suite. Its
Status block explicitly deferred Tasks 3 and 4 with the markers
Plan 03 → 04 and playback Plan 06 → 07 precedent: scaffold +
property suite ships standalone so the FSM/runner shape can be
reviewed in isolation, then the ~200-call-site App carve-out
lands on its own.

After this sprint the narrow `#[allow(dead_code)]` attributes on
`search_fsm::Input`, `search_fsm::Output`, `SearchFsm`,
`SearchRunner`, and `TypedAction` all come off — every type has a
live caller.

## Tasks

### 1. Keybinding extraction (Plan 08 Task 3)

Today `src/ui/actions.rs` conflates two roles:
- `crossterm::KeyEvent` → `Action` (config-driven keymap)
- `Action` → App-method dispatch (imperative call inside
  `App::dispatch`)

Split into:
- **Keymap layer** (already partially present as
  `actions::Keymap`): `KeyEvent → Option<Action>`. Pure lookup,
  stdio-core-swappable.
- **FSM adapter layer** (new): `Action → Option<search_fsm::Input>`.
  Pure function. Every search-relevant Action maps to exactly one
  Input; non-search Actions map to `None` (routed to other
  sub-FSMs in later sprints).

Two pure functions, two separate test suites. stdio-core's form
protocol can bypass the keymap layer and inject Actions (or
Inputs) directly.

### 2. App integration (Plan 08 Task 4)

Replace the five bare fields on `App` with
`search_runner: SearchRunner`. Every call-site that reads or
writes those fields routes through `SearchRunner` accessors and
`SearchRunner::dispatch`.

Expected call-site count: ~200 (per Plan 08's estimate). Includes
the main dispatch loop, the debounce tick, the similarity-mode
toggle (`--similar PATH` startup), the preview hook, and
widget-layer reads.

The mechanical pattern follows markers Plan 04:
- Add `search_runner: SearchRunner` field to `App`.
- Migrate one field at a time (`search_in_progress`,
  `search_pending`, `query_changed`, `in_similarity_mode`,
  `similarity_results`), committing between each for
  `cargo test` green.
- Each migration: remove the field + initialiser; add an
  accessor method; convert read/write call-sites.

### 3. Wire the runner into the event loop

`App::dispatch(Action)` for search-relevant Actions now calls
`SearchRunner::dispatch(Input)` and honors the returned
`DispatchResult`:

```rust
let res = self.search_runner.dispatch(input);
match res.output {
    Some(Output::SpawnSearch { query }) => self.spawn_search_task(query),
    Some(Output::CancelSearch) => self.abort_search_task(),
    Some(Output::FilterSimilarity { query }) => {
        let warn = self.search_runner.apply_similarity_filter(&query);
        if let Some(msg) = warn { self.set_status(msg.to_string()); }
    }
    Some(Output::FireSelection) => {
        if let Some(action) = res.action {
            self.dispatch_typed_action(action);
        }
    }
    None => {}
}
```

The App owns the tokio task handle (`Option<JoinHandle<()>>`)
and performs spawn/abort. FSM stays I/O-free. When task batches
arrive on mpsc, App calls `set_results` + `append_results` +
`set_total_matches` and fires `Input::SearchStarted` (first
batch) / `SearchSettled { total }` (completion) /
`SearchFailed` (task error before any batch).

### 4. `TypedAction` dispatch

Route `TypedAction::LoadSample(path)` to the existing
shell-open logic (or whatever the current `OpenSelected` Action
does). New method `App::dispatch_typed_action(TypedAction)` for
future-proofing against additional variants.
