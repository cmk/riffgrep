## Goal

Port marker-selection, bank, sync, and visibility state into an explicit
`rust-fsm` state machine and validate its invariants with
`proptest-state-machine`.

## Context

Markers is the first of three regions being ported to `rust-fsm` (see
`doc/designs/debt-fsm.md` for the roadmap; playback and TUI follow in
separate sprints). Motivation: 7 fix commits across SPRINT7/11/12 plus
`3fa73fc` / `6d23741` / `46168e6`, all clustering on implicit
state-coordination failures between `App.selected_marker`,
`App.active_bank`, `App.bank_sync`, `App.markers_visible`, and the nested
`MarkerBank` in `Preview`.

Current state surface (`src/ui/mod.rs:126-275`):
- `selected_marker: Option<usize>` — which of {SOF, M1, M2, M3} is selected
- `active_bank: Bank { A, B }` — which bank edits target
- `bank_sync: bool` — mirror edits across banks
- `markers_visible: bool` — guard flag: blocks edits when false
- `preview.markers: Option<MarkerConfig { bank_a: MarkerBank, bank_b: MarkerBank }>`

Coverage today: ~55 unit tests, zero proptests.

### 2. FSM module `src/engine/marker_fsm.rs`

> **Note (2026-04-18):** The sketch below is a pre-implementation draft
> and differs from what shipped. The authoritative API is in
> `src/engine/marker_fsm.rs` itself; the shipped implementation uses
> `StateMachineImpl` (not the `state_machine!` macro), stores banked
> marker data in `MarkerConfig` (not a generic `BankPair<T>`), has 19
> `Input` variants (not 23 — the Small/Large/Snap variants collapsed
> into a single delta-carrying `NudgeForward(u32)` / `NudgeBackward(u32)`
> / `SetSelectedMarker(u32)`), and returns `Option<Output>` from
> `consume` (visibility-gated no-ops return `Some(unchanged)` + `None`
> output rather than a dedicated `Noop` variant). Task 4b later added
> `Input::LoadConfig(MarkerConfig)` for preview-load sync. Future FSM
> sprints should read the source, not this sketch.

**Problem:** Marker state is ad-hoc fields on `App`; transitions are
scattered across `set_marker`, `select_next_marker`, `nudge_marker`,
`toggle_bank`, `toggle_bank_sync`, `marker_reset`, `clear_nearest_marker`,
`clear_bank_markers`, `toggle_marker_display`
(`src/ui/mod.rs:1170-2185`), each with its own guard logic.

**Solution:** A `state_machine!` definition with:

- **State struct:** `MarkerFsmState { selection: Selection, active_bank: Bank, bank_sync: bool, visible: bool, banks: BankPair<MarkerBank> }`
  - `Selection { Sof, M1, M2, M3 }`
  - `Bank { A, B }`
  - `BankPair<T> { a: T, b: T }`
- **Input enum (23 variants):** `SetMarker1/2/3`, `ToggleBank`,
  `ToggleBankSync`, `SelectNextMarker`, `SelectPrevMarker`,
  `ClearNearestMarker`, `ClearBankMarkers`, `NudgeForward{Small,Large}`,
  `NudgeBackward{Small,Large}`, `SnapZeroCrossing{Forward,Backward}`,
  `MarkerReset`, `IncrementRep`, `DecrementRep`, `ToggleMarkerDisplay`,
  `ExportMarkersCsv(PathBuf)`, `ImportMarkersCsv(PathBuf)`.
- **Output enum:** side-effect descriptors (`Output::Noop`,
  `Output::DidEdit`, `Output::WroteCsv(PathBuf)`) — lets tests assert on
  effects without running I/O.
- **Transition rules:** implement per `debt-fsm.md` invariants. Enforce
  `visible == false` ⇒ all edit inputs become `Noop`. Enforce
  `bank_sync == true` ⇒ mutations apply to both banks.

**API surface:**
- `pub struct MarkerFsm { machine: StateMachine<MarkerBankMachine> }`
- `pub fn consume(&mut self, input: Input) -> Output`
- `pub fn state(&self) -> &MarkerFsmState`
- Getter methods matching the current App API surface so UI read-sites
  don't change (`selected_marker()`, `active_bank()`, `bank_sync()`,
  `markers_visible()`, `marker_value(Selection, Bank)`).

### 3. Carve out existing App state

**Problem:** UI code reads/mutates the four bare fields directly.

**Solution:** Replace the four fields on `App` with `marker_fsm: MarkerFsm`.
Update every call-site in `src/ui/mod.rs` and `src/ui/actions.rs` to
dispatch through `self.marker_fsm.consume(...)`. Preserve the existing
`App` public API (no behavior change observable externally). The nested
`preview.markers` data moves into the FSM state as
`banks: BankPair<MarkerBank>`.

**API surface:** `App` loses 4 fields, gains 1. Expected dispatch
call-sites: ~40 (grep `self.selected_marker`, `self.active_bank`,
`self.bank_sync`, `self.markers_visible`).

### 4. Property-test layout `tests/engine/marker_fsm/`

**Problem:** No property tests today; invariants in `debt-fsm.md` are
not verified.

**Solution:** Create the four-file layout:

- `tests/engine/marker_fsm/gen.rs` — action generators using `prop_oneof`.

  `src/ui/mod.rs` and `src/engine/marks.rs`, focused on regressions
  `3fa73fc`, `6d23741`, `46168e6`, SPRINT12 F1/F2, SPRINT11, SPRINT7 F7.
- `tests/engine/marker_fsm/test.rs` — integration entrypoint declaring
  all four modules and any shared fixtures.

### 5. Properties (invariants from `debt-fsm.md`)

**Must-pass:**

| #  | Property                                                                                                                                     |
|----|----------------------------------------------------------------------------------------------------------------------------------------------|
| P1 | `MarkerReset` is idempotent: `reset ∘ reset ≡ reset` in all states                                                                           |
| P2 | `MarkerReset` from SOF-selected initial state is a no-op                                                                                     |
| P3 | `SelectPrev ∘ SelectNext ≡ id` except at M3-selected                                                                                         |
| P4 | `SelectNext ∘ SelectPrev ≡ id` except at SOF-selected                                                                                        |
| P5 | `NudgeForward ∘ NudgeBackward ≡ id` for markers not at SOF/EOF (both Small and Large variants)                                               |
| P6 | Bank-sync preservation: initial `bank_sync=true` + stream excludes `ToggleBankSync` ⇒ `banks.a == banks.b` at every step                     |
| P7 | Markers-disabled fixed point: after `ToggleMarkerDisplay` flips visible to false, no edit input changes state until another `ToggleMarkerDisplay` |
| P8 | No-op sub-sequence insertion: inserting `[SelectPrev, SelectNext]` into a stream (at any position where neither is at its boundary) preserves the end state |

**Deferred (document with `#[ignore]` + TODO if they fail, fix in
follow-up sprint):**
- Any property whose failure reveals a logic bug requiring more than a
  transition-table adjustment.
